# Техническое задание
## Платформа «Женское общество» — защищённая версия

**Версия документа:** 1.0
**Назначение:** ТЗ для разработчика (или команды), который заново соберёт приложение с повышенными требованиями к безопасности, приватности и соответствию 152-ФЗ.

Документ описывает: что это за продукт, какие роли и функции, как устроены данные, и — главное — какие требования к безопасности нужно выполнить. Текущий работающий прототип используется как эталон функциональности; раздел 11 перечисляет его известные слабые места, которые новая версия должна закрыть.

---

## 1. Назначение и контекст

«Женское общество» — закрытое женское онлайн-сообщество (PWA / веб-приложение, приоритет — мобильный iOS Safari). Ядро продукта:

- **методика саморазвития** через «Колесо баланса» (6 сфер жизни + центр «Состояние»), мягкая диагностика, дневник, женский цикл, «путь» из конкретных шагов;
- **закрытый клуб** с материалами, событиями и экспертами;
- **монетизация:** платформа бесплатна для участниц; эксперты платят за доступ к платформе и продают свои продукты (мастермайнды, консультации, эфиры, группы) с удержанием комиссии платформы **15%**;
- **бережный тон, без «зависимых» механик** (никаких стриков, очков, рейтингов активности).

Аудитория — женщины, РФ. Обрабатываются персональные данные → обязательное соответствие **152-ФЗ «О персональных данных»**.

## 2. Роли пользователей

| Роль | Назначение | Ключевые права |
|---|---|---|
| **Участница** (member) | Основной пользователь. Платформа бесплатна | Диагностика, колесо, дневник, цикл, чтение материалов/событий, покупка продуктов экспертов, чаты, бадди |
| **Эксперт** (mentor) | Ведёт и продаёт контент | Своя страница-витрина, публикация материалов (через модерацию), создание мероприятий/мастермайндов, продажи, баланс. Платит за доступ к платформе |
| **Куратор** (curator) | Ведёт группы (в развитии) | Управление группами, заданиями, встречами |
| **Администратор / Владелец** (admin) | Модерация и управление платформой | Модерация контента, дашборд платформы (люди/контент/активность/деньги), назначение ролей |

**Критично для безопасности:** роль должна определяться и проверяться **только на сервере**. В прототипе есть клиентское переключение ролей («тестовый режим») — в продакшене его быть не должно (см. 11.1).

## 3. Функциональные требования (модули)

### 3.1 Онбординг и методика
- Регистрация/вход, онбординг новой участницы, входная диагностика («знакомство с собой»): ресурс, агентность, выбор сфер, опорная сфера, маркеры, шкала благополучия.
- **Колесо баланса:** 6 сфер (тело, близкие, дом, финансы, дело, отдых) + центр «Состояние» = среднее из самочувствия и оценок сфер. Оценки 0–10, история динамики «было → стало».
- До 3 фокус-сфер. Цель по сфере → декомпозиция на «шаги пути» (разовые/повторяющиеся).
- **Дневник состояния** (текст, настроение, теги, привязка к сфере), голосовой ввод.
- **Женский цикл:** даты, симптомы, подсказка о фазе (ориентир, не медицина).
- Прогресс/уровни растут от реальных отметок маркеров, **не** от активности в приложении.

### 3.2 Контент клуба
- **Материалы** трёх форматов: статья / видео / аудио. Ссылка или загрузка файла; встроенные плееры (YouTube, Rutube, VK, Vimeo, Дзен, Яндекс Музыка, SoundCloud, прямой файл, PDF).
- **Общая база материалов + модерация:** эксперт публикует → статус «на модерации» → администратор одобряет/отклоняет (с причиной) → одобренные видят все. Автор видит статус.
- Сохранение (лайк = сохранить), отклики/комментарии, автообложки.

### 3.3 Эксперты
- Страница-витрина эксперта (специализация, о себе, темы, контакты), каталог экспертов.
- Кабинет: список материалов со статусами, мероприятия, мастермайнды, баланс «Заработано на платформе».
- Личные чаты 1:1 (участница ↔ эксперт).

### 3.4 События и платные продукты
- Мероприятия (онлайн/офлайн, дата, места, цена).
- **Мастермайнды** — платные продукты эксперта (видны всем, оплата через платформу).
- (В развитии) другие типы платного контента: эфиры, группы/групповая терапия, консультации, доступ к информации.

### 3.5 Сообщество
- Поиск «бадди» (напарница по сфере), приглашения, статусы.
- Чаты 1:1, общие/групповые обсуждения, уведомления.

### 3.6 ИИ-ассистент
- Бережный помощник на базе **российской LLM (YandexGPT)**; отдельная «память» (тред) на каждую сферу и на «состояние»; кликабельные ссылки на материалы/экспертов в ответах.

### 3.7 Монетизация
- **Доступ эксперта к платформе** (платно, тарифы по функционалу — в развитии).
- **Продажи продуктов эксперта** участницам; сплит: **85% эксперту, 15% платформе** с каждой продажи.
- Баланс эксперта; история платежей.

### 3.8 Кабинет владельца (дашборд)
- Показатели в реальном времени: **Люди** (участниц, экспертов, новых за 7 дней), **Контент** (материалов/на модерации/опубликовано), **Активность** (активных за 7 дней, записей в дневник, отмеченных шагов), **Деньги** (оборот, доля платформы, начислено экспертам, активные подписки, число оплат).
- Все сводные показатели считаются на сервере, **без доступа к личным данным** участниц.

## 4. Модель данных (сущности)

Минимальный набор сущностей (в новой версии — нормализованные таблицы, **не** один JSONB-блоб на пользователя):

- **users / profiles** — аккаунт, роль, ПДн (имя, город, возраст и т.д.) — хранить ПДн отдельно от поведенческих данных.
- **diagnostics, sphere_scores, sphere_score_history, sphere_goals, path_steps** — методика.
- **journal_entries** — дневник (чувствительные данные).
- **cycle_data** — цикл (данные о здоровье — особая категория ПДн, повышенная защита).
- **materials** (+ status, author, moderation-поля), **material_comments**.
- **events**, **masterminds**, **event_registrations**.
- **expert_profiles**.
- **buddy_requests, buddy_invites**.
- **messages** (личные чаты).
- **assistant_threads** — история диалогов с ИИ (чувствительно).
- **subscriptions / access** — доступ эксперта к платформе.
- **payments / transactions** — платежи, сплит, статусы; **ledger** для сверки балансов.
- **admins / roles** — привилегии.
- **audit_log** — журнал действий (модерация, платежи, смена ролей).

> Дневник, цикл, диалоги с ИИ, данные диагностики — **чувствительные**. Требуют строжайшего разграничения доступа и (желательно) отдельного шифрования.

## 5. Требования к безопасности (основной раздел)

### 5.1 Аутентификация
- Политика паролей (минимум 10–12 символов, проверка на утёкшие пароли), хеширование (Argon2/bcrypt).
- **Обязательное подтверждение e-mail**; защита от перебора: rate-limiting, блокировка после N попыток, CAPTCHA при подозрении.
- Опциональная **двухфакторная аутентификация** (email/TOTP) — минимум для ролей эксперт/куратор/администратор.
- Безопасные сессии: короткоживущие access-токены + refresh, ротация, возможность отзыва; по возможности httpOnly/SameSite cookies. Выход завершает сессию на сервере.
- Восстановление пароля по одноразовой ссылке с TTL; уведомление на почту о входах/сменах.

### 5.2 Авторизация (RBAC)
- **Роли и права — только на сервере.** Клиент не может выдать себе роль.
- Принцип наименьших привилегий; **deny-by-default**: доступ к данным закрыт, пока явно не разрешён.
- Разграничение на уровне строк и по возможности колонок для каждой таблицы. Каждый запрос проверяет «это его данные / у него есть право».
- Привилегированные действия (модерация, платежи, смена ролей, дашборд владельца) — за серверной проверкой роли + журналируются.
- Убрать любые клиентские «тестовые» переключатели ролей из продакшена.

### 5.3 Защита данных
- **Шифрование в транзите** (TLS 1.2+ везде) и **at rest** (шифрование БД и хранилища).
- **Минимизация ПДн:** собирать только необходимое; чувствительные поля (цикл/здоровье, дневник) — с повышенной защитой, отдельным доступом, при возможности — прикладным шифрованием.
- **Право на удаление** (забвение): полное удаление аккаунта и связанных данных по запросу; политика хранения и авто-удаления.
- **Управление согласиями:** явное согласие на обработку ПДн при регистрации, версия политики, дата; экспорт своих данных по запросу.

### 5.4 Валидация ввода и защита от инъекций
- Серверная валидация **всех** входных данных (типы, длины, диапазоны, whitelisting).
- Параметризованные запросы (нет строковой склейки SQL).
- **Экранирование вывода** и санитайзинг пользовательского контента (тексты материалов, комментарии, профиль) — защита от XSS. Никакого небезопасного рендера HTML.
- Валидация и нормализация ссылок на медиа (защита от SSRF/встраивания вредоносного контента).

### 5.5 Загрузка файлов
- Приватное хранилище по умолчанию; доступ по **подписанным URL** с TTL, без публичного листинга.
- Проверка типа/размера/содержимого, **антивирус-сканирование**, ограничение форматов.
- Отдельный домен/бакет для пользовательских файлов; заголовки, запрещающие исполнение.

### 5.6 Ограничение частоты и защита от злоупотреблений
- Rate-limiting на аутентификацию, ИИ, платежи, публикацию, отправку сообщений.
- WAF / защита от ботов; лимиты на объём и частоту загрузок.

### 5.7 Аудит и мониторинг
- **Журнал аудита** для действий модерации, платежей, смены ролей, доступа к чувствительным данным (кто, что, когда).
- Мониторинг безопасности, алерты на аномалии, логирование ошибок без утечки ПДн/секретов.

### 5.8 Управление секретами
- Секреты (ключи ИИ, платёжные ключи, сервисные токены) — только на сервере, в защищённом хранилище (vault/секреты окружения), **никогда** в клиентском коде и репозитории; регулярная ротация.

### 5.9 Модерация и безопасность контента
- Модерация **всего** пользовательского контента (материалы, мастермайнды, комментарии, профили).
- Механизм жалоб/репортов и блокировки пользователей; защита от спама.

## 6. Приватность и 152-ФЗ

- **Локализация ПДн граждан РФ на территории РФ** (ст. 18 152-ФЗ): БД с персональными данными — на серверах в РФ (Yandex Cloud / VK Cloud / SberCloud или иной РФ-хостинг). Зарубежные облака для ПДн недопустимы.
- ИИ-обработка — **российская модель (YandexGPT)**; в модель не передаются лишние ПДн; данные не уходят за пределы РФ.
- Регистрация в реестре операторов ПДн (уведомление РКН), политика конфиденциальности, пользовательское соглашение, согласия.
- Особая категория (данные о здоровье — цикл) — усиленная защита и отдельное согласие.
- Опубликованный клубный контент (не ПДн) может храниться на общем хостинге; личные данные — только в РФ-контуре.

## 7. Платежи (требования безопасности)

- Использовать **сертифицированного платёжного провайдера РФ** (ЮKassa, CloudPayments, Продамус, СБП), а не самописную обработку карт. Минимизировать PCI-DSS-скоуп (хостируемая форма/редирект, токенизация) — карточные данные не проходят через наш сервер.
- **Суммы и сплит считаются на сервере**; клиент **никогда** не передаёт сумму/долю. Комиссия платформы (15%) и доля эксперта (85%) вычисляются серверно.
- **Подтверждение оплаты — через webhook провайдера с проверкой подписи**, не по «успешному» ответу клиента. Идемпотентность (защита от двойных списаний/двойных зачислений).
- Разделение выплат: через сплитование провайдера **или** внутренний реестр (ledger) с последующей выплатой и сверкой; хранить статусы, возвраты, комиссии.
- Полный **аудит платежей**, возвраты/refund, отчётность. Балансы эксперта и платформы — из подтверждённых транзакций.
- Учитывать налоговый/юридический контур (самозанятые/ИП эксперты, договоры-оферты).

> В текущем прототипе платежи — **заглушки** (имитация), логика сплита готова, но реального провайдера, webhook-подтверждения и защиты от подделки сумм нет. Это обязательный пункт новой версии.

---

## 8. ИИ-ассистент (требования безопасности)

- Вызов ИИ — **только для аутентифицированных пользователей** (проверка токена на стороне серверной функции); публичный незащищённый эндпоинт недопустим.
- **Лимиты по пользователю** и общий бюджет (защита от злоупотреблений и неконтролируемых расходов).
- Секретный ключ модели — только на сервере.
- Митигация **prompt-injection**; фильтрация/минимизация ПДн в промпте; отказ от передачи чувствительных данных третьим сторонам.
- Логирование обращений без хранения лишних ПДн; данные не покидают РФ-контур.

> В прототипе серверная функция ИИ публична (вызов по URL без авторизации) — в новой версии обязательно закрыть авторизацией и лимитами.

## 9. Нефункциональные требования

- **Платформа:** адаптивный веб/PWA, приоритет мобильного iOS Safari; офлайн-устойчивость PWA по возможности.
- **Производительность:** быстрый первичный рендер, разумный размер бандла (код-сплиттинг), отзывчивость на слабой сети.
- **Доступность (a11y)** и русскоязычный интерфейс, бережный тон, отсутствие «зависимых» механик.
- **Надёжность:** резервное копирование БД и хранилища, план восстановления (DR), мониторинг доступности.
- **Наблюдаемость:** логи, метрики, алерты (без утечки ПДн).

## 10. Рекомендации по стеку и архитектуре

Стек — на усмотрение разработчика, но с соблюдением принципов ниже.

- **Frontend:** SPA/PWA (например, React + TypeScript). Никакой бизнес-логики доверия на клиенте.
- **Backend:** выделенный серверный слой (например, Node.js/NestJS, или любой на выбор) с **сервер-авторитарной** бизнес-логикой. Клиент вызывает API, сервер проверяет права, валидирует, считает суммы.
- **БД:** PostgreSQL в **РФ-облаке**; разграничение доступа на уровне строк/колонок; нормализованная схема (не единый JSONB-блоб на пользователя).
- **Хранилище файлов:** приватное объектное хранилище с подписанными URL.
- **ИИ:** YandexGPT через защищённую серверную функцию.
- **Платежи:** интеграция с РФ-провайдером через webhooks.
- **Инфраструктура:** секреты в vault, CI с проверкой зависимостей и статическим анализом, разделение сред (dev/stage/prod).

> Допустимо использовать managed-BaaS (как Supabase), но **только в self-hosted/РФ-варианте** для ПДн и с корректно настроенными политиками доступа. Ключевой принцип остаётся: сервер — источник правды по правам, суммам и ролям.

## 11. Известные слабые места текущего прототипа (закрыть в новой версии)

1. **Клиентское переключение ролей** («тестовый режим»): UI-роль меняется на клиенте. Реальные привилегии частично защищены серверной проверкой администратора, но сам переключатель в проде недопустим — роли назначаются только сервером/администратором.
2. **Единый JSONB-блоб** всех данных пользователя: грубая гранулярность доступа. Нужна нормализация и разграничение по чувствительности.
3. **Платежи — заглушки:** нет реального провайдера, webhook-подтверждения, защиты сумм от подделки, идемпотентности.
4. **ИИ-функция публична** (без авторизации и лимитов).
5. **Публичное файловое хранилище** без сканирования и подписанных URL.
6. **ПДн в зарубежном облаке** (прототип на внешнем BaaS) — несоответствие 152-ФЗ по локализации.
7. **Нет журнала аудита, MFA, политики паролей, антибот-защиты, механизма жалоб/блокировок, права на удаление данных** — добавить.
8. **Модерация только материалов** — распространить на все типы пользовательского контента.

## 12. Критерии приёмки (чек-лист безопасности)

- [ ] Роли и права проверяются на сервере; клиент не может повысить привилегии.
- [ ] Deny-by-default доступ к данным; разграничение по строкам/колонкам на всех таблицах.
- [ ] Подтверждение e-mail, политика паролей, rate-limiting входа, MFA для привилегированных ролей.
- [ ] ПДн граждан РФ хранятся в РФ; оформлены согласия и политика конфиденциальности; работает удаление данных.
- [ ] Платежи через сертифицированного провайдера; суммы/сплит считаются на сервере; подтверждение по webhook с подписью; идемпотентность; аудит.
- [ ] ИИ-эндпоинт закрыт авторизацией и лимитами; ключи только на сервере.
